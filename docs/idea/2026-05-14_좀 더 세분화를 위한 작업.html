가능합니다. 오히려 이번 지적은 기획이 잘못됐다기보다, 핵심을 더 날카롭게 만들 기회에 가깝습니다. 현재 기획서에서도 이미 “관심사 기반 개인화 트렌드 큐레이션 앱”이라는 방향과, 정보 분산·관심사 불일치·맥락 부족·탐색 피로라는 문제 정의는 잡혀 있습니다. 다만 교수님 입장에서는 아직 이렇게 보였을 가능성이 큽니다.

“트렌드 앱인 건 알겠는데, 왜 굳이 이 앱이어야 하지?” “관심사 기반이라고 했는데, 관심사를 얼마나 세밀하게 다룰 수 있지?” “커스터마이징이 단순 카테고리 선택인지, 추천 시스템의 핵심인지 불명확하다.”

그래서 다음 발표에서는 커스터마이징 = 이 앱의 필요성을 증명하는 핵심 기능으로 잡는 게 좋습니다.

1. 문제의 본질: “트렌드가 많다”가 아니라 “나에게 맞지 않는다”

지금까지의 필요성은 약간 이렇게 들릴 수 있습니다.

“요즘 트렌드가 많고 흩어져 있으니 모아서 보여준다.”

이건 맞는 말이지만, 교수님이 보기에는 다른 앱이나 포털도 할 수 있는 말입니다. 조금 더 깊게 들어가면 문제는 다음과 같습니다.

기존 문제 더 구체화한 문제
트렌드 정보가 많다 사용자가 관심 없는 트렌드까지 봐야 해서 탐색 시간이 길어진다
카테고리가 넓다 같은 엔터테인먼트라도 아이돌, 영화, 웹툰, 예능 관심사가 다르다
인기 키워드만 보인다 왜 유행하는지, 내 관심사와 무슨 관련이 있는지 알기 어렵다
여러 플랫폼을 봐야 한다 검색, SNS, 커뮤니티 흐름을 사용자가 직접 비교해야 한다

즉, 앱의 필요성을 이렇게 바꾸는 게 좋습니다.

이 앱은 단순히 최신 트렌드를 보여주는 앱이 아니라, 사용자의 세부 관심사를 기준으로 불필요한 트렌드를 줄이고, 필요한 트렌드의 맥락을 빠르게 이해하도록 돕는 개인화 트렌드 필터링 앱이다.

여기서 핵심 단어는 최신성보다 선별성입니다.

2. 커스터마이징을 단순 설정이 아니라 “추천 기준 설정”으로 정의하기

커스터마이징을 그냥 “사용자가 관심 카테고리를 고른다”로 설명하면 약합니다. 발표에서는 다음처럼 정의하는 게 더 좋습니다.

커스터마이징이란 사용자가 대분류, 세부분야, 관심 키워드, 비선호 키워드를 설정하여 자신에게 맞는 트렌드 추천 기준을 직접 조정하는 기능이다.

즉, 단순히 패션 / 음식 / IT / 엔터테인먼트를 고르는 게 아니라, 아래처럼 깊이가 있어야 합니다.

단계 예시 의미
1단계: 대분류 엔터테인먼트 넓은 관심 영역
2단계: 세부분야 아이돌, 영화, 드라마, 웹툰, 예능 사용자의 실제 관심 분야
3단계: 관심 키워드 뉴진스, 넷플릭스, 로맨스 드라마, 웹툰 원작 추천 정확도를 높이는 직접 키워드
4단계: 비선호 설정 정치 이슈 제외, 스포츠 제외, 특정 키워드 숨김 불필요한 추천 감소
5단계: 활동 기반 보정 클릭, 저장, 검색 기록 사용 행동을 바탕으로 추천 가중치 조정

이렇게 설명하면 커스터마이징이 단순 UI가 아니라 DB, 추천 로직, API 명세와 연결되는 핵심 기능이 됩니다.

3. 카테고리는 “대분류 + 세부분류 + 태그” 구조가 가장 적절합니다

교수님이 말씀하신 “엔터테인먼트도 더 세분화할 수 있지 않나?”라는 지적은 꽤 중요합니다. 현재 6개 카테고리만 있으면 너무 넓습니다.

추천 구조

대분류 세부분류 예시 키워드/태그 예시
엔터테인먼트 아이돌, 영화, 드라마, 예능, 웹툰 컴백, 티저, 넷플릭스, OST, 원작
스포츠 야구, 축구, 농구, e스포츠 KBO, EPL, LOL, 경기결과, 이적
게임 모바일게임, PC게임, 콘솔게임, e스포츠 업데이트, 신작, 패치, 랭킹, 굿즈
음식 맛집, 신메뉴, 디저트, 편의점, 카페 신상, 한정판, 레시피, 지역맛집
패션 스트릿, 미니멀, 브랜드, 계절코디 봄코디, 스니커즈, 브랜드콜라보
뷰티 스킨케어, 메이크업, 향수, 헤어 쿠션, 립틴트, 올리브영, 퍼스널컬러
IT/디지털 AI, 스마트폰, 앱, 디바이스 신제품, 업데이트, 리뷰, 비교

발표에서는 이렇게 말하면 좋습니다.

“기존에는 6개 대분류만 고려했지만, 교수님 피드백을 반영하여 실제 사용자 관심사는 더 세밀하다고 판단했습니다. 따라서 대분류-세부분류-키워드 태그 구조로 관심사를 재설계했습니다.”

이 문장이 꽤 좋습니다. 피드백을 반영했다는 느낌도 있고, ERD/API로 연결하기도 쉽습니다.

4. 커스터마이징 설계 대안 2가지

대안 A. 고정 카테고리 기반

사용자가 미리 정해진 대분류와 세부분류를 선택하는 방식입니다.

장점 단점
구현이 쉽고 발표용 ERD가 명확함 사용자의 세부 취향을 완전히 반영하기 어려움
카테고리 관리가 안정적임 새로운 유행어, 신조어 반영이 느릴 수 있음
초기 캡스톤 구현에 적합함 개인화 수준이 약해 보일 수 있음

예시:

엔터테인먼트 → 아이돌 / 영화 / 드라마 / 웹툰 선택

대안 B. 태그/키워드 기반

사용자가 직접 관심 키워드를 추가하거나, 앱이 추천 키워드를 제안하는 방식입니다.

장점 단점
개인화 느낌이 강함 DB와 추천 로직이 조금 복잡해짐
최신 유행어 대응이 쉬움 키워드 중복, 오타, 유사어 처리가 필요함
“커스터마이징”을 강조하기 좋음 초기 구현 범위가 커질 수 있음

예시:

사용자가 “야구”, “KBO”, “한화”, “이적시장” 키워드를 직접 추가

추천안: 하이브리드 방식

캡스톤 발표와 구현 가능성을 고려하면 고정 카테고리 + 관심 키워드 추가 방식이 가장 좋습니다.

구성 설명
대분류 패션, 음식, IT/디지털, 엔터테인먼트, 뷰티, 게임, 스포츠 등
세부분류 엔터테인먼트 안의 아이돌/영화/드라마/웹툰 등
관심 키워드 사용자가 직접 추가하거나 추천 키워드에서 선택
비선호 키워드 관심 없는 주제 숨김
활동 기반 보정 클릭/저장/검색 기록으로 추천 우선순위 조정

이렇게 가면 발표에서 논리가 깔끔합니다.

“초기에는 카테고리 기반으로 안정적인 추천을 제공하고, 이후 사용자의 관심 키워드와 활동 데이터를 반영하여 개인화를 고도화합니다.”

5. 필요성 구체화는 “사용자 상황”으로 보여주는 게 좋습니다

필요성을 추상적으로 말하지 말고, 실제 사용자 상황으로 보여주면 설득력이 강해집니다.

발표용 사용자 시나리오 예시

사용자 문제 상황 앱이 해결하는 방식
A 사용자 야구를 좋아하지만 전체 스포츠 뉴스에서 축구·농구 이슈까지 섞여 나옴 스포츠 → 야구 → KBO 관심사를 설정해 야구 관련 트렌드 우선 제공
B 사용자 아이돌 트렌드만 보고 싶은데 엔터테인먼트 전체 이슈가 너무 넓음 엔터테인먼트 → 아이돌 세부분류 선택, 관련 키워드 중심 추천
C 사용자 유행 키워드는 봤지만 왜 유행하는지 모름 AI 요약으로 유행 이유, 관련 키워드, 출처를 함께 제공
D 사용자 여러 플랫폼을 직접 확인하기 번거로움 앱에서 관심사 기반 트렌드를 통합 조회

이런 식으로 보여주면 교수님이 묻는 “왜 필요한가?”에 대한 답이 됩니다.

6. ERD에 반영해야 할 방향

다음 발표에 ERD가 들어간다면, 커스터마이징이 DB 구조에 드러나야 합니다. 단순히 users, trends만 있으면 개인화 앱처럼 보이지 않습니다.

예시 테이블은 이렇게 잡을 수 있습니다.

테이블 예시 역할
users 사용자 정보 저장
interest_categories 대분류/세부분류 카테고리 저장
user_interests 사용자가 선택한 관심사 저장
user_interest_keywords 사용자가 직접 추가한 관심 키워드 저장
user_block_keywords 사용자가 제외한 비선호 키워드 저장
trends 수집된 트렌드 키워드 저장
trend_categories 트렌드와 카테고리 연결
user_trend_actions 클릭, 저장, 조회 등 사용자 행동 기록
trend_ai_analysis AI 요약, 유행 이유, 관련 키워드 저장

특히 interest_categories는 이런 구조가 좋습니다.

필드 예시 설명
category_id 카테고리 ID
parent_id 상위 카테고리 ID
category_name 카테고리명
depth 대분류/세부분류 구분
is_active 사용 여부

이렇게 하면 엔터테인먼트 → 아이돌, 스포츠 → 야구 같은 계층 구조를 표현할 수 있습니다.

7. API 명세서와 연결하는 방식

API 명세도 커스터마이징 흐름 중심으로 잡으면 좋습니다.

기능 API 예시 설명
카테고리 목록 조회 GET /categories 대분류/세부분류 목록 조회
내 관심사 저장 POST /users/me/interests 사용자가 선택한 관심사 저장
내 관심사 수정 PATCH /users/me/interests 관심사 추가/삭제
관심 키워드 추가 POST /users/me/keywords 직접 입력한 관심 키워드 저장
비선호 키워드 추가 POST /users/me/block-keywords 추천에서 제외할 키워드 저장
개인화 트렌드 조회 GET /trends/recommendations 관심사 기반 트렌드 목록 반환
트렌드 상세 조회 GET /trends/{trendId} 키워드 설명, AI 요약, 관련 키워드 조회
사용자 행동 저장 POST /trends/{trendId}/actions 클릭, 저장, 조회 기록 반영

여기서 중요한 건 API를 많이 만드는 게 아니라, 발표에서 이렇게 말하는 것입니다.

“커스터마이징에서 저장된 관심사와 키워드는 추천 API의 기준값으로 사용됩니다. 따라서 사용자가 설정을 바꾸면 추천 트렌드 목록도 함께 달라집니다.”

이 한 문장이 데이터 흐름을 잘 설명합니다.

8. 발표자료에 넣기 좋은 핵심 문장

아래 문장은 거의 그대로 PPT에 넣어도 괜찮습니다.

기존 트렌드 서비스는 전체 인기 키워드를 중심으로 제공되기 때문에,
사용자는 자신의 관심사와 관련 없는 정보까지 함께 확인해야 한다.

본 서비스는 사용자의 대분류, 세부분류, 관심 키워드, 비선호 키워드를 기반으로
개인화된 트렌드 목록을 제공하여 탐색 피로를 줄이고,
AI 요약을 통해 트렌드의 유행 이유와 관련 키워드를 빠르게 이해할 수 있도록 돕는다.

조금 더 짧게 쓰면:

본 앱의 핵심 필요성은 “최신 트렌드 제공”이 아니라
“사용자 관심사에 맞는 트렌드 선별과 맥락 이해 지원”이다.

9. 다음 발표 구성 추천

다음 발표는 이 순서가 가장 안정적입니다.

순서 발표 내용 핵심 메시지
1 기존 문제 트렌드는 많지만 사용자 관심사와 맞지 않는다
2 필요성 구체화 관심 없는 정보 제거, 세부 관심사 반영, 맥락 제공이 필요하다
3 커스터마이징 구조 대분류-세부분류-키워드-비선호 설정
4 추천 흐름 관심사 저장 → 트렌드 필터링 → AI 요약 → 개인화 목록 제공
5 ERD 사용자 관심사와 트렌드 데이터를 어떻게 연결하는지 제시
6 API 명세 관심사 저장/수정/추천 조회 API 제시
7 기대 효과 탐색 시간 감소, 관심사 적합도 향상, 트렌드 이해도 향상

요소 설명
