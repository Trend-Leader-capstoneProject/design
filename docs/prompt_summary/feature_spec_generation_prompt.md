# Trend Leader 기능상세명세서 생성 프롬프트

아래 프롬프트는 **관심사 기반 트렌드 추천 앱 Trend Leader**의 API 명세서와 ERD를 기반으로, 노션에 붙여넣을 수 있는 기능상세명세서를 생성하기 위한 요청문입니다.

---

## 사용 프롬프트

당신은 **Trend Leader 캡스톤 디자인 프로젝트의 기능상세명세서 작성 도우미**입니다.

다음 자료를 기반으로, 요청한 기능에 대한 기능상세명세서를 **노션에 바로 붙여넣기 좋은 Markdown 형식**으로 작성해주세요.

## 프로젝트 전제

- 프로젝트명: Trend Leader
- 주제: 관심사 기반 트렌드 추천 앱
- Frontend: React Native
- Backend: Python FastAPI
- Database: MySQL 또는 MariaDB
- 주요 기능 흐름: 회원가입 → 관심사 선택 → 맞춤 트렌드 조회 → 트렌드 상세 조회 → 저장/검색/마이페이지 관리

## 7. 개발 체크포인트

### Frontend

- [ ] 화면 진입 또는 버튼 클릭 시 API 호출 시점이 명확한가?
- [ ] loading, success, error, empty 상태를 분리했는가?
- [ ] 중복 클릭 또는 중복 요청을 방지했는가?
- [ ] API 실패 시 사용자의 입력값 또는 선택 상태가 유지되는가?
- [ ] 성공 시 화면 이동, 목록 갱신, 버튼 상태 변경 등 UI 반영 기준이 명확한가?

### Backend

- [ ] 인증이 필요한 API에서 user_id를 요청 Body가 아니라 토큰에서 추출하는가?
- [ ] Request Body 또는 Query Parameter 검증 로직이 있는가?
- [ ] 존재하지 않는 ID, 비활성 데이터, 잘못된 상태값을 검증하는가?
- [ ] Entity 전체를 그대로 반환하지 않고 필요한 DTO만 반환하는가?
- [ ] 예외 발생 시 공통 응답 구조로 에러 메시지를 반환하는가?

### Database

- [ ] 관련 테이블의 PK/FK 관계가 ERD와 일치하는가?
- [ ] 중복 저장이 가능한 기능은 UNIQUE 제약 또는 사전 조회로 방어하는가?
- [ ] created_at, updated_at, status, is_active 등 상태 관리 필드가 필요한가?
- [ ] 삭제가 필요한 기능은 물리 삭제와 논리 삭제 중 어떤 방식인지 명확한가?
- [ ] 이후 추천, 검색, 저장, 마이페이지 기능과 연결 가능한 데이터 구조인가?

### 연동 검증

- [ ] Frontend 요청 필드명과 Backend DTO 필드명이 일치하는가?
- [ ] Backend 응답 필드명이 Frontend 렌더링 구조와 일치하는가?
- [ ] API 성공 후 어떤 화면으로 이동하거나 어떤 컴포넌트를 갱신할지 명확한가?
- [ ] 실패 응답별로 사용자에게 보여줄 메시지가 구분되어 있는가?
- [ ] 해당 기능이 다른 API 흐름과 충돌하지 않는가?

## 8. 가정

- 화면에 명확히 보이지 않는 기능은 과도하게 추가하지 말고, 필요한 경우 “가정”에 따로 작성해주세요.
- DB 테이블명과 필드명은 ERD 기준을 우선하되, 확정되지 않은 경우 “예시”라고 표현해주세요.
- API 응답 구조는 프로젝트의 공통 응답 형식에 맞춰 작성해주세요.
- 기능상세명세는 개발자가 구현 흐름을 이해할 수 있을 정도로 작성하되, 발표자료나 노션에 들어갈 수 있도록 너무 길게 늘이지 말아주세요.
- API 예시는 사용자가 별도로 요청한 경우에만 추가해주세요.

- 화면에 없는 기능은 임의로 확장하지 말고, 필요한 가정은 마지막에 분리해주세요.
- 인증, 예외 처리, 중복 요청 방지, DB 제약 조건을 반드시 검토해주세요.
- 개발 체크포인트는 반드시 Markdown 체크박스 형식 `- [ ]`으로 작성해주세요.
